
## miden::core::crypto::dsa::ecdsa_k256_keccak
| Procedure | Description |
| ----------- | ------------- |
| verify | Verifies an secp256k1 ECDSA signature compatible with `miden-crypto::ecdsa_k256_keccak`.<br /><br />This wrapper mirrors the materialization performed in `miden-crypto::ecdsa_k256_keccak`: given<br />a public key commitment and the original message, it reconstructs the calldata expected by the<br />precompile (public key bytes, Keccak256(message), signature). The public key and signature are<br />supplied via the advice stack, and can be obtained with the `ecdsa_k256_keccak` function.<br /><br />Inputs:<br />Operand stack: [PK_COMM, MSG, ...]<br />Advice stack:  [PK[9] \| SIG[17] \| ...]<br />Outputs:<br />Operand stack: []<br />Advice stack:  []<br /><br />Where:<br />- `PK_COMM`: Poseidon2 hash commitment of the 32-byte ECDSA public key<br />- `MSG`: single word (4 field elements) representing the message to verify<br />- `PK[9]`: 33-byte public key packed as 9 field elements on advice stack<br />- `SIG[17]`: 65-byte signature packed as 17 field elements on advice stack<br /><br />Local memory layout (element addresses):<br />- locaddr[0 ..9 ] : compressed public key (33 bytes packed as 9 felts)<br />- locaddr[12..20] : message bytes (MSG written as eight u32 limbs)<br />- locaddr[20..28] : keccak256(message) digest (8 felts)<br />- locaddr[28..45] : signature (65 bytes packed as 17 felts)<br /><br />The procedure traps if:<br />- The public key does not hash to `PK_COMM` (invalid commitment)<br />- The signature verification fails<br /> |
| verify_prehash | Verifies an ECDSA signature with pre-hashed message using deferred execution.<br /><br />This procedure is intended for manual signature verification where the caller<br />has already computed the message digest.<br /><br />The caller provides the public key, the pre-hashed message digest, and the signature data in<br />memory. This routine forwards the request to the host precompile and returns the boolean result.<br />In typical flows the digest is obtained from `keccak256::hash_bytes`, but any 32-byte prehash<br />is accepted.<br /><br />Input:  `[pk_ptr, digest_ptr, sig_ptr, ...]`<br />Output: `[result, ...]`<br /><br />Where:<br />- `pk_ptr`: word-aligned memory address containing the 33-byte compressed secp256k1 public key<br />- `digest_ptr`: word-aligned memory address containing the 32-byte message digest<br />- `sig_ptr`: word-aligned memory address containing the 65-byte signature<br />- `result`: 1 if the signature is valid, 0 if invalid<br /><br />All data must be stored in memory as packed u32 values (little-endian), with unused bytes<br />in the final u32 set to zero.<br /> |
| verify_prehash_native | Verifies an ECDSA secp256k1 signature in the VM. Same calldata format as `verify_prehash`:<br />pk is 33 BE bytes, digest is 32 BE bytes, sig is 65 BE bytes (r \|\| s \|\| v), all packed<br />as u32-LE felts.<br /><br />Stack: [pk_ptr, digest_ptr, sig_ptr, ...]  ->  [result, ...]<br /><br />Returns 1 if the signature is valid, 0 otherwise. No trap on failure: every check<br />contributes to a multiplicative validity flag, and the math is run with safe-fallback<br />values whenever an input would otherwise trap a sub-proc.<br /><br />Validation:<br />- pk parses (prefix in {0x02, 0x03}) and decompresses to an on-curve point Q.<br />- r in (0, n) and s in (0, n).<br />- R = u1*G + u2*Q is not the identity, where u1 = e * s^-1 mod n, u2 = r * s^-1 mod n.<br />- R.x mod n == r.<br /><br />Local memory (144 felts):<br />0..8     : x_pk<br />8        : parity<br />12..32   : Q (decompressed pk)<br />32..40   : r<br />40..48   : s<br />48..56   : e (digest as integer)<br />76..96   : R = u1*G + u2*Q<br />96..104  : s_inv<br />104..112 : u1<br />112..120 : u2<br />128      : validity flag (initialised to 1, multiplicatively ANDed with each check)<br />129..132 : pk_ptr / digest_ptr / sig_ptr caches<br /> |
| verify_prehash_native_precomp | Same contract as `verify_prehash_native`, except `R = u1*G + u2*Q` is computed via<br />`k1_point::verify_precomp` against a Merkle-committed joint comb table fetched per-window<br />from the advice stack. The caller passes only the `pk_aug` commitment; the<br />compressed PK, the Merkle root over the table, and the per-window entries are all<br />witness data fetched from the advice provider.<br /><br />Stack: [pk_aug_word (4 felts), digest_ptr, sig_ptr, ...]  ->  [result, ...]<br /><br />`pk_aug` is the 4-felt augmented public-key commitment, defined as<br />pk_aug = Poseidon2::hash_elements(compressed_PK_9 \|\| zeros_3 \|\| merkle_root_4)<br />The 3 zero felts pad the 9-felt compressed PK to a 12-felt boundary so the hash input is<br />aligned to 16 felts. See "Soundness boundary" below for where `pk_aug` must come from.<br /><br />Advice preconditions (host responsibility):<br />- Advice map: `pk_aug -> [compressed_PK_9, zeros_3, merkle_root_4]` (16 felts).<br />- Advice stack: 32 padded entries (24 felts each = 768 felts total), in window<br />order (LSB-first), selected by the verifier's per-window indices into the joint<br />comb table for `(G, Q)` at width `w = 8`.<br />- Merkle store: contains authentication paths for those 32 entries in the depth-21<br />table tree. Each leaf is `Poseidon2::hash_elements(padded_entry_24)`.<br /><br />Soundness boundary. The VM proc binds `pk_aug` to `(compressed_PK, merkle_root)` via<br />the augmented-PK hash check, but it does NOT prove that `merkle_root` is the comb table<br />for `decompress(compressed_PK)`. A caller that can choose `pk_aug` freely could commit a<br />root for a different point and still pass verification against that other point.<br /><br />Account-bound `pk_aug` (sound). In the protocol's account model, `pk_aug` lives in<br />account storage and is transitively bound to the AccountId via the storage commitment.<br />The account creator's commitment is the trust anchor -- the same trust assumption the<br />protocol already operates under for `Poseidon2(PK)`. Self-custody, multi-sig, delegated<br />keys, cross-account verification, and recursive composition all fall in this category.<br /><br />Caller-supplied `pk_aug` (NOT sufficient alone). For open-recovery flows (e.g. verify a<br />MetaMask sig from an arbitrary address) no pre-registered `pk_aug` exists, and a<br />validity bit alone is meaningless. Such flows must additionally check the recovered<br />point against an out-of-band identifier (e.g. an Eth address derived from the recovered<br />PK), which this proc does not return. Use a separate entry point for that case.<br /><br />No-trap behavior. Within the packed-u32 calldata contract (i.e. all calldata felts are<br />valid u32 values), semantically invalid inputs return 0 without trapping: a wrong<br />`pk_aug`, an unprepared `pk_aug` (the advice-map lookup is gated on `adv.has_mapkey`),<br />a non-zero preimage pad, a malformed digest or signature payload, an off-curve PK, a<br />non-canonical r or s, or an ECDSA equation that doesn't hold. Outside that contract --<br />a caller that writes non-u32 felts into the calldata memory regions -- the u32* ops in<br />`_parse_be_u256`, `_parse_sig`, and `_parse_compressed_pk` will trap.<br /><br />The proc IS NOT no-trap against host-misconfigured witness:<br />1. A pushed advice entry whose hash doesn't match the leaf at the corresponding<br />tree index traps `verify_precomp`'s `mtree_verify`. Avoiding the trap would require<br />a Poseidon2 second-preimage on the committed leaf (the honest entry that hashes to<br />that leaf is fixed by the published `merkle_root`, so the attacker must find a<br />different 24-felt entry that hashes to the same value).<br />2. `point_add` traps on entries with non-canonical X/Y limbs that survive the leaf<br />check.<br />3. Fewer than 768 entry felts on the advice stack traps `adv_pipe` mid-stream.<br />4. After `adv.push_mapval`, the next 16 advice felts must be the augmented-PK<br />preimage. The helper supplies this as a 16-felt map value. A different map length<br />changes how that 16-felt prefix is sourced and can shift subsequent comb-table<br />advice; non-u32 felts in the compressed-PK region trap `_parse_compressed_pk`.<br />All four are host bugs (the prover failed to set up its own witness consistently); they<br />are not reachable by any caller-controlled input alone.<br /><br />Local memory layout matches `verify_prehash_native`. Additional slots:<br />136..145 : compressed_PK scratch (9 felts, streamed from advice map by adv_pipe)<br />145..148 : zero pad scratch (3 felts, streamed from advice map)<br />148..152 : merkle_root scratch (4 felts, word-aligned, streamed from advice map)<br />`pk_aug` stays on the operand stack across the binding check rather than going to a<br />cache slot.<br /> |
| verify_keccak_native_precomp | Verifies an ECDSA signature after hashing a 32-byte message word with Keccak-256,<br />using an augmented public key that commits to the precomputed comb table.<br /><br />This is the precomputed-comb counterpart to `verify`: the message word is first<br />hashed with `keccak256_native::keccak256_bytes`, then<br />`verify_prehash_native_precomp` checks the signature against the augmented public-key<br />commitment.<br /><br />Inputs:<br />Operand stack: [MSG, PK_AUG, ...]<br />Advice stack:  [SIG[17] \| precomp_witness \| ...]<br />Outputs:<br />Operand stack: [valid, ...]<br /><br />Where:<br />- `MSG`: single word (4 felts). Each felt is split into two little-endian u32 limbs,<br />yielding the 32-byte Keccak input at `mem[0..8]`.<br />- `PK_AUG`: 4-felt augmented public-key commitment<br />`Poseidon2(compressed_PK_9 \|\| 0, 0, 0 \|\| merkle_root)`. See<br />`verify_prehash_native_precomp` for the trust boundary.<br />- `SIG[17]`: 65-byte signature `(r, s, v)` packed as 17 little-endian u32 felts on advice.<br />- `precomp_witness`: the per-call witness consumed by `verify_prehash_native_precomp`<br />(advice-map entry at key `pk_aug` plus the comb-window entries on the advice stack).<br />- `valid`: 1 if the signature is valid under `pk_aug` against `Keccak-256(MSG)`; else 0.<br /> |
