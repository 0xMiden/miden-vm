
## miden::core::collections::mmr
| Procedure | Description |
| ----------- | ------------- |
| get | Loads the leaf at the absolute `pos` in the MMR.<br /><br />This MMR implementation supports only u32 positions.<br /><br />The position must refer to an existing leaf: `pos < num_leaves`, where `num_leaves` is stored<br />at `mmr_ptr[0]`. The procedure fails instead of returning an empty or sentinel word when `pos`<br />is outside the current MMR.<br /><br />Stack transition:<br />Input: [pos, mmr_ptr, ...]<br />Output: [N, ...] where `N` is the leaf and `R` is the MMR peak that owns the leaf.<br /> |
| num_leaves_to_num_peaks | Given the num_leaves of a MMR returns the num_peaks.<br /><br />Implemented as counting the number of "1" bits in `num_leaves`.<br /><br />Input: [num_leaves, ...]<br />Output: [num_peaks, ...]<br />Cycles: 67<br /> |
| num_peaks_to_message_size | Given the num_peaks of a MMR, returns the hasher state size after accounting<br />for the required padding.<br /><br />Input: [num_peaks, ...]<br />Output: [len, ...]<br />Cycles: 19<br /> |
| root | Computes the raw rooted frontier root of the given MMR.<br /><br />This root does not bind the MMR length by itself. When exposed across trust boundaries,<br />it must be authenticated together with the MMR length stored at `mmr_ptr`.<br /><br />The root folds the path to `num_leaves`: each set bit consumes a peak on the left, while each<br />unset bit folds in an empty subtree on the right. Empty-subtree roots are derived on the fly by<br />squaring (`merge(empty, empty)`), starting from the empty leaf `ZERO`.<br /><br />The accumulator and current empty-subtree root are kept on the operand stack as two words<br />`[ACC, EMPTY]`, with the remaining length bits in `loc.0` and the peak cursor in `loc.1`, so the<br />fold avoids per-iteration memory traffic. Two folding shortcuts keep the hash count minimal:<br />- leading unset bits are folded once instead of twice, since `ACC` and `EMPTY` coincide until<br />the first peak is consumed;<br />- `EMPTY` is squared only while an unset bit still remains, skipping a trailing all-ones run.<br /><br />Input: [mmr_ptr, ...]<br />Output: [ROOT, ...]<br /> |
| root_with_len | Computes the authenticated rooted frontier pair for the given MMR.<br /><br />The returned root is the raw frontier root from `root`; the MMR length is returned alongside it<br />so callers can authenticate `(num_leaves, root)`.<br /><br />Input: [mmr_ptr, ...]<br />Output: [ROOT, num_leaves, ...]<br /> |
| mmb_root | Computes the Merkle Mountain Belt prototype root.<br /><br />The MMB memory layout is `[num_leaves, 0, 0, 0] \|\| peaks`, where the peaks are U-MMB<br />mountains ordered from oldest to newest.<br /><br />The range boundaries are not tracked via mountain heights: with `bits = num_leaves + 1`,<br />the splits sit at the set bits of `split_mask = bits & !((bits << 1) & (bits >> 1))` (see<br />`mmb_verify_leaf`). The mask is shifted so the boundary after the oldest mountain sits at<br />bit 31, and the fold consumes one bit per peak from the top â€” the bit is the exit condition<br />of the inner per-range loop, so the fold needs no per-mountain branch at all.<br /><br />The range and belt accumulators are kept on the operand stack as `[RANGE_ACC, BELT_ACC]`,<br />so the fold avoids per-iteration memory traffic; only the boundary bits and the peak cursor<br />bound live in locals.<br /><br />Input: [mmb_ptr, ...]<br />Output: [ROOT, ...]<br /> |
| mmb_root_with_len | Computes the authenticated `(num_leaves, MMB_ROOT)` pair.<br /><br />Input: [mmb_ptr, ...]<br />Output: [ROOT, num_leaves, ...]<br /> |
| mmb_verify_leaf | Verifies a Merkle Mountain Belt prototype leaf proof against authenticated `(num_leaves, ROOT)`.<br /><br />The proof is stored at `proof_ptr` using this memory layout:<br />- `proof_ptr + 0`: proof length<br />- `proof_ptr + 1`: opened position<br />- `proof_ptr + 4`: opened leaf word<br />- then one 8-element entry per sibling:<br />- `entry + 0`: direction, where 0 means sibling is left and 1 means sibling is right<br />- `entry + 4`: sibling word<br /><br />The proof is bound to the opened position: the MMB shape is fully determined by the<br />authenticated `num_leaves`, so the mountain containing `position`, its height, and the<br />range/belt fold counts are derived here in closed form (no loop over the mountains) and the<br />proof length and every stored direction are asserted equal to that canonical path. A proof<br />for one position therefore cannot be relabeled to authenticate another.<br /><br />With `bits = num_leaves + 1` (the shape bit pattern `mmb_root` folds over) and<br />`g = num_leaves - position`, the closed forms are:<br />- mountain bit position: `pos = m - 1 + (bits mod 2^m + 2^m <= g)` where `m = ilog2(g)`;<br />- mountain height: `pos + ((bits >> pos) & 1)`; mountain start: `((bits >> (pos+1)) - 1)<br />shifted left by `pos+1`, so the in-mountain position is `position - start`;<br />- range splits sit at the set bits of `split_mask = bits & !((bits << 1) & (bits >> 1))`<br />(a split between the mountains at bit positions `k` and `k-1` for each set bit `k`), so<br />with `low = split_mask & (2^(pos+1) - 2)`: the ranges right of the leaf's range are<br />`popcount(low)` and the peaks right of its mountain in its range are `pos - ilog2(low \| 1)`.<br /><br />Input: [proof_ptr, ROOT, num_leaves, ...]<br />Output: [...]<br /> |
| mmb_unpack | Authenticates an MMB commitment `(num_leaves, ROOT)` and loads its peaks into memory.<br /><br />The peaks are provided via the advice map keyed by `ROOT`, encoded as `[num_leaves, 0, 0, 0]`<br />followed by the peaks (oldest to newest, zero-padded to a 16-word minimum and an even count,<br />the same padding rule `pack`/`unpack` use). The peaks are loaded into memory at `mmb_ptr` in<br />the `mmb_root` layout and the belt root is recomputed over them and asserted equal to `ROOT`.<br /><br />`num_leaves` is taken from the operand stack as a *trusted* input and is what the recomputed<br />belt is folded over, which authenticates the advice peaks against the `(num_leaves, ROOT)`<br />pair. Note that unlike the rooted frontier â€” whose root folds over the length bits â€” the raw<br />belt root does NOT bind the length by itself: lengths with the same mountain count and range<br />structure (e.g. 7 and 8) fold the same peaks to the same root. The MMB commitment is<br />therefore the pair, as in the crypto-side `BeltSummary`; callers must never treat `ROOT`<br />alone as a commitment.<br /><br />On success the MMB is laid out at `mmb_ptr` exactly as `mmb_root` expects, so `mmb_add` can<br />be used against the authenticated peaks.<br /><br />Input: [ROOT, num_leaves, mmb_ptr, ...]<br />Output: [...]<br /> |
| mmb_add | Appends a leaf to the MMB.<br /><br />This is the lazy U-MMB append: the new leaf joins as a height-0 mountain and at most the<br />rightmost mergeable pair of mountains is merged. With `m = num_leaves' + 1` (the post-append<br />shape bits), the merge is skipped exactly when `m` is a power of two; otherwise the pair<br />sits at mountain positions `j` and `j + 1` from the right, where `j` is the index of the<br />lowest set bit of `m`, and the mountains right of the pair slide down one slot.<br /><br />This updates the peaks in memory only; the belt commitment is recomputed on demand by<br />`mmb_root`.<br /><br />Input: [EL, mmb_ptr, ...]<br />Output: [...]<br /> |
| unpack_frontier | <br />The peaks are provided via the advice map keyed by `ROOT`, encoded as `[num_leaves, 0, 0, 0]`<br />followed by the peaks (the same value layout `pack`/`unpack` use). The peaks are loaded into<br />memory at `mmr_ptr` and the frontier root is recomputed over them and asserted equal to `ROOT`.<br /><br />`num_leaves` is taken from the operand stack as a *trusted* input (committed alongside `ROOT`)<br />and is what the recomputed root is folded over, so the pair `(num_leaves, ROOT)` is bound<br />together: a longer frontier padded with empty leaves shares the raw root but is rejected here.<br /><br />On success the MMR is laid out at `mmr_ptr` exactly as `unpack` leaves it, so `get`/`add` can be<br />used against the authenticated peaks.<br /><br />Input: [ROOT, num_leaves, mmr_ptr, ...]<br />Output: [...]<br /> |
| unpack | Writes the MMR who's peaks hash to `HASH` to the memory location pointed to by `mmr_ptr`.<br /><br />Input: [HASH, mmr_ptr, ...]<br />Output: [...]<br /><br />Where:<br />- HASH: is the MMR peak hash, the hash is expected to be padded to an even<br />length and to have a minimum size of 16 elements<br />- The advice map must contain a key with HASH, and its value is<br />`[num_leaves, 0, 0 , 0] \|\| hash_data`, and hash_data is the data used to computed `HASH`<br />- mmr_ptr: the memory location where the MMR data will be written to,<br />starting with the MMR forest (its total leaves count) followed by its peaks.<br />The address is expected to be word-aligned.<br /><br />Cycles: 162 + 9 * extra_peak_pair cycles<br />where `extra_peak` is the number of peak pairs in addition to the first<br />16, i.e. `round_up((num_of_peaks - 16) / 2)`<br /> |
| pack | Computes the hash of the given MMR and copies it to the Advice Map using its hash as a key.<br /><br />Input: [mmr_ptr, ...]<br />Output: [HASH, ...]<br />Cycles: 128 + 3 * num_peaks<br /> |
| add | Adds a new element to the MMR.<br /><br />This will update the MMR peaks in the VM's memory and the advice provider<br />with any merged nodes.<br /><br />Input: [EL, mmr_ptr, ...]<br />Output: [...]<br />Cycles: 145 + 39 * peak_merges<br /> |
