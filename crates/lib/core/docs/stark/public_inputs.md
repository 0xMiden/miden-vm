
## miden::core::stark::public_inputs
| Procedure | Description |
| ----------- | ------------- |
| process_public_inputs | Processes the public inputs.<br /><br />This is a generic dispatch procedure that calls the instance-specific implementation<br />via `dynexec`. The instance-specific implementation populates the FLPI section of the ACE<br />READ region with the fixed-length public inputs and absorbs every public input into the<br />Fiat-Shamir transcript in a rate-aligned schedule (mirroring the Rust-side `MultiAir::observe`).<br />The reduced variable-length public inputs (the `reduced_inputs` block) are staged by the<br />instance's `verify_proof` before the transcript starts; this procedure only absorbs them.<br /><br />Memory layout populated (low -> high addresses):<br /><br />reduced_inputs (16 felts, at a fixed standalone address, NOT in the ACE READ region)<br />[..., pi (FLPI EF, m base felts -> 2m slots),<br />beta0, beta1, alpha0, alpha1, <pad>, OOD-start, ...]<br /><br />where:<br />a) `reduced_inputs` holds [kernel_H (4) \| program_digest (4) \| transcript_state (4) \|<br />zero_pad (4)] — the caller-facing summary of the variable-length public inputs.<br />`kernel_H` is the hash of all kernel-procedure digests; `program_digest` and<br />`transcript_state` are passed through unchanged. The trailing pad rate-aligns the FS<br />observation. This block lives at a fixed standalone address (see `stark/constants.masm`),<br />not contiguous with `pi`.<br />b) `pi` holds the fixed-length public inputs (stack i/o) as extension field elements.<br />Each base felt sits at `pi + 2*i`, with zero in the high coordinate.<br />c) `[beta0, beta1, alpha0, alpha1]` is the auxiliary randomness (loaded<br />non-deterministically, verified in step II).<br />d) `<pad>` is the codegen alignment word; OOD-start is the first felt of the OOD<br />evaluations section.<br /><br />The variable-length inputs are also folded into a running `c_total` accumulator (using<br />the non-deterministically supplied alpha/beta), stored at `C_TOTAL_PTR` for the<br />outer-LogUp boundary check in step II.<br /><br />Inputs:  [...]<br />Outputs: [...]<br /><br />Invocation: exec<br /> |
| compute_and_store_public_inputs_address | Computes the addresses where the public inputs are to be stored.<br /><br />In order to call `eval_circuit`, we need to lay out the inputs to the constraint evaluation<br />circuit in a contiguous region of memory (called READ section in the ACE chiplet documentation)<br />right before the region storing the circuit description (called EVAL section).<br /><br />As the number of public inputs is a per-instance parameter, while the sizes of the OOD<br />evaluation frames and the number of auxiliary random values are fixed, we lay out the public<br />inputs right before the auxiliary random values and OOD evaluations.<br />Hence the address where public inputs are stored is computed using a negative offset from the<br />address where the OOD evaluations are stored.<br /><br />We compute one pointer:<br />pi_ptr = OOD_EVALUATIONS_PTR - 8 - num_fixed_len_pi * 2<br /><br />The `* 2` converts the FLPI base-felt count to EF slots (each base felt is stored as<br />`[val, 0]`). The `-8` skips the two words between FLPI and OOD: the auxiliary randomness word<br />and the codegen alignment pad. The reduced-inputs block is no longer carved out of this region;<br />it lives at the fixed standalone `REDUCED_INPUTS_PTR` (see `stark/constants.masm`), and this<br />procedure publishes that fixed address into the reduced-inputs pointer cell.<br /><br />Inputs:  [num_fixed_len_pi, ...]<br />Outputs: [...]<br /><br />Invocation: exec<br /> |
| load_base_store_extension_double_word_from_mem | Reads 8 base field elements from caller memory at `src`, stores them as extension field elements<br />at `dst`, and returns the original base words in the rate positions for absorption by the caller<br />via `poseidon2::permute`.<br /><br />The procedure does NOT absorb the elements into the transcript -- the caller is responsible<br />for calling `poseidon2::permute` after this procedure to complete the absorption.<br /><br />Inputs:  [Y, Y, C, dst, src, ...]<br />Outputs: [A0, A1, C, dst + 16, src + 8, ...]<br /><br />where A0 and A1 are the original base element words (from memory), positioned in the rate slots<br />so that a subsequent `poseidon2::permute` absorbs them into the sponge state.<br /><br />Invocation: exec<br /> |
