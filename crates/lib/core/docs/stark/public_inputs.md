
## miden::core::stark::public_inputs
| Procedure | Description |
| ----------- | ------------- |
| process_public_inputs | Processes the public inputs.<br /><br />This is a generic dispatch procedure that calls the instance-specific implementation<br />via `dynexec`. The instance-specific implementation loads the fixed-length public inputs<br />and the variable-length public inputs from the advice stack, populating the caller-facing<br />reduced-inputs block and the FLPI section of the ACE READ region, and absorbing every<br />public input into the Fiat-Shamir transcript in a rate-aligned schedule (mirroring the<br />Rust-side `MultiAir::observe`).<br /><br />Memory layout populated (low -> high addresses):<br /><br />[..., reduced_inputs (16 felts), pi (FLPI EF, m base felts -> 2m slots),<br />beta0, beta1, alpha0, alpha1, <pad>, OOD-start, ...]<br /><br />where:<br />a) `reduced_inputs` holds [kernel_H (4) \| zero_pad (4) \| program_digest (4) \|<br />transcript_state (4)] — the caller-facing summary of the variable-length public<br />inputs. `kernel_H` is the Poseidon2 hash of all kernel-procedure digests;<br />`program_digest` and `transcript_state` are passed through unchanged.<br />b) `pi` holds the fixed-length public inputs (stack i/o) as extension field elements.<br />Each base felt sits at `pi + 2*i`, with zero in the high coordinate.<br />c) `[beta0, beta1, alpha0, alpha1]` is the auxiliary randomness (loaded<br />non-deterministically, verified in step II).<br />d) `<pad>` is the codegen alignment word; OOD-start is the first felt of the OOD<br />evaluations section.<br /><br />The variable-length inputs are also folded into a running `c_total` accumulator (using<br />the non-deterministically supplied alpha/beta), stored at `C_TOTAL_PTR` for the<br />outer-LogUp boundary check in step II.<br /><br />Inputs:  [...]<br />Outputs: [...]<br /><br />Invocation: exec<br /> |
| compute_and_store_public_inputs_address | Computes the addresses where the public inputs are to be stored.<br /><br />In order to call `eval_circuit`, we need to lay out the inputs to the constraint evaluation<br />circuit in a contiguous region of memory (called READ section in the ACE chiplet documentation)<br />right before the region storing the circuit description (called EVAL section).<br /><br />As the number of public inputs is a per-instance parameter, while the sizes of the OOD<br />evaluation frames and the number of auxiliary random values are fixed, we lay out the public<br />inputs right before the auxiliary random values and OOD evaluations.<br />Hence the address where public inputs are stored is computed using a negative offset from the<br />address where the OOD evaluations are stored.<br /><br />We compute two pointers:<br />pi_ptr             = OOD_EVALUATIONS_PTR - 8 - num_fixed_len_pi * 2<br />reduced_inputs_ptr = pi_ptr - 16<br /><br />The `* 2` for FLPI converts the base felt count to EF slots (each base felt is stored as<br />`[val, 0]`). The `-8` accounts for the 2 words between FLPI end and OOD start:<br />- aux_rand word `[beta0, beta1, alpha0, alpha1]` at AUX_RAND_ELEM_PTR (4 felts);<br />- codegen alignment padding (4 felts) — the ACE codegen MASM layout policy double-word-<br />aligns the main-trace OOD region, which forces a 1-word gap after the word-aligned<br />randomness slot.<br />The `-16` reserves the 16-felt caller-facing reduced-inputs block immediately below FLPI:<br />`[kernel_H \| zero_pad \| program_digest \| transcript_state]`. The zero pad keeps the FS<br />observation rate-aligned (8-felt chunks) when `kernel_H` and the single-word VLPI messages<br />are absorbed.<br /><br />The reduced-inputs block is NOT consumed by the ACE circuit; it sits there as a stable<br />caller-readable summary of the variable-length public inputs after `verify_proof` returns.<br /><br />Inputs:  [num_fixed_len_pi, ...]<br />Outputs: [...]<br /><br />Invocation: exec<br /> |
| load_base_store_extension_double_word | Loads 8 base field elements from the advice stack, stores them as extension field elements<br />in memory, and returns the original base element words in the rate positions for absorption<br />by the caller via `poseidon2::permute`.<br /><br />The procedure does NOT absorb the elements into the transcript -- the caller is responsible<br />for calling `poseidon2::permute` after this procedure to complete the absorption.<br /><br />Inputs:  [Y, Y, C, ptr, ...]<br />Outputs: [A0, A1, C, ptr + 16, ..]<br /><br />where A0 and A1 are the original base element words (from advice), positioned in the rate<br />slots so that a subsequent `poseidon2::permute` absorbs them into the sponge state.<br /><br />Invocation: exec<br /> |
