| point_negate | Negation: out = -P = (X, -Y, is_infinity).<br /><br />Stack:<br />[out_addr, p_addr, ...]  ->  [...]<br /><br />For a finite point (X, Y), produces (X, p - Y) via `f_k1::neg`. | point_add | Addition: out = P1 + P2.<br /><br />Stack:<br />[out_addr, p1_addr, p2_addr, ...]  ->  [...]<br /><br />Branches on the four cases:<br />1. | msm_4_128_glv_k1 | WARNING -- NOT A GENERIC MSM. The scan uses a<br />16-entry LUT, with `LUT[3] = P_1 + P_2` hardcoded to one of 4 constants. Calling this with<br />`P_1` or `P_2` not in their permitted sets silently produces a WRONG result; the proc<br />never checks the precondition.<br /><br />Pre-condition (callers MUST honor):<br />P_1 ∈ {G, -G},     P_2 ∈ {φ(G), -φ(G)},     P_3 ∈ {Q, -Q},     P_4 ∈ {φ(Q), -φ(Q)},<br />where G is the secp256k1 generator and φ((x, y)) = (β·x mod p, y) is the endomorphism.<br />The only intended caller is `verify_glv_hinted`.<br /><br />`sign_a_u1` and `sign_b_u1` are the u_1 sign bits from `k1_scalar::verify_glv_split`.<br />Used to select the LUT[3] = P_1 + P_2 constant, since P_1 + P_2 = (±G) + (±φ(G)) takes<br />one of 4 fixed values.<br /><br />`lut12_endo` is a u1 flag for the special case `P_4 = φ(P_3)` after the caller<br />has formed the signed P_3 and P_4 inputs. In `verify_glv_hinted`, where<br />`P_3 = ±Q` and `P_4 = ±φ(Q)`, this condition is exactly<br />`sign_a_u2 == sign_b_u2`.<br /><br />When the flag is 1, LUT[12] = P_3 + P_4 is computed as<br />`P_3 + φ(P_3) = -φ²(P_3)`. Since β is a nontrivial cube root of unity<br />(`1 + β + β² = 0`) and `P_4.x = β·P_3.x`, the result is<br />`x = -(P_3.x + P_4.x)`, `y = -P_3.y`, and<br />`is_infinity = P_3.is_infinity`.<br /><br />The proc does not verify the endomorphism relation. Passing `1` for arbitrary<br />P_3, P_4 makes LUT[12] incorrect.<br /><br />Each scalar `k_i` must fit in 128 bits (4 u32 LE limbs at `k_i_addr..k_i_addr+4`).<br />Each base point `P_i` follows the standard 20-felt affine layout (X, Y, is_infinity, 3<br />reserved zeros). | verify_glv_hinted | Computes `R = u_1·G + u_2·Q` for the secp256k1 generator `G`, the public-key point `Q`,<br />and two 256-bit scalars `(u_1, u_2)`. This is the inner relation an ECDSA verifier needs<br />to check; the caller does the final `R.x mod n == r` comparison.<br /><br />Why this proc is faster than a straight 256-iter scalar mult. secp256k1 has an<br />"endomorphism" `φ(x, y) = (β·x mod p, y)` — a cheap map (one base-field multiplication)<br />that, when applied to any curve point `P`, has the effect of scalar multiplication by a<br />specific 256-bit integer `λ`: that is, `φ(P) = [λ]P`. Using that, any 256-bit scalar `k`<br />can be split into a pair of 128-bit signed integers `(k_a, k_b)` satisfying<br />`k ≡ k_a + k_b·λ (mod n)`, so `[k]P = [k_a]P + [k_b]·φ(P)`. The host computes the split<br />(`glv_split_k1` event handler) using Babai rounding on a precomputed lattice basis; the<br />VM verifier (`k1_scalar::verify_glv_split`) checks the relation modulo `n` and the<br />magnitude bounds. Splitting both `u_1` and `u_2` rewrites our equation as a<br />4-scalar 128-bit MSM:<br />R = k_a_u1·G + k_b_u1·φ(G) + k_a_u2·Q + k_b_u2·φ(Q)<br />which `msm_4_128_glv_k1` evaluates in 128 doubling iterations rather than 256. Sign bits<br />from the GLV decomposition are absorbed by negating the corresponding base point in place.<br />The result is written to the caller's `out_addr` in the standard 20-felt affine layout.<br /><br />Stack transition:<br />[out_addr, q_addr, u_2_addr, u_1_addr, ...]  ->  [...]<br /><br />Local memory layout (mem[0..128]):<br />0..20    : G        (hardcoded constant; then in-place ±-negated -> P_1)<br />20..40   : φ(G)     (hardcoded constant: (β·G_x mod p, G_y); ±-negated -> P_2)<br />40..60   : Q        (copied from q_addr; then in-place ±-negated -> P_3)<br />60..80   : φ(Q)     (computed: β·Q_x, Q_y, Q.is_infinity; then in-place ±-negated -> P_4)<br />80..98   : verify_glv_split witness for u_1 (mag_a [8] + mag_b [8] + sign_a [1] + sign_b [1])<br />100..118 : same for u_2 (offset 100 keeps mag_a word-aligned)<br />120..124 : cached (out_addr, u_1_addr, u_2_addr, q_addr)<br /> |
| verify_precomp | Computes `R = [u_1] * G + [u_2] * Q` against a Merkle-committed joint comb table for<br />`(G, Q)` at width `w = 8`. The host provides Merkle authentication paths and pushes the<br />32 entries (in window order, 24 felts per entry = 768 felts total) onto the advice stack.<br /><br />Per window the driver pulls one entry into a local scratch slot, hashes it, runs<br />`mtree_verify` against the Merkle root, and adds the entry to the running accumulator<br />via `point_add`. `256 / 8 = 32` windows, one `point_add` per window; no<br />`point_double`.<br /><br />The driver processes one u32 limb from each scalar at a time. Each limb supplies four<br />byte windows; each byte is peeled with `u32divmod.256`.<br /><br />Caller responsibility:<br />- The advice stack must hold the 32 selected entries for `(u_1, u_2)` in window order<br />(768 felts total), derived from the cached precomputed public key and pushed via<br />`AdviceInputs::with_stack`.<br />- `merkle_root_addr` must point at the 4-felt root, and that root must already be bound to<br />`Q` via an outer commitment (e.g., the augmented-PK design used by<br />`ecdsa_k256_keccak::verify_prehash_native_precomp`).<br />- The host's Merkle store must contain authentication paths for the 32 selected entries<br />under the depth-21 root at `merkle_root_addr`.<br /><br />Trap profile: this proc is NOT no-trap.<br />1. `mtree_verify` traps when the pushed entry doesn't hash to the leaf at the<br />computed index under the given root -- i.e., when the pushed entries don't<br />match the committed tree. An honest host always satisfies this; avoiding the trap<br />would require a Poseidon2 second-preimage on the committed leaf (the honest entry<br />that hashes to that leaf is fixed by the published `merkle_root`, so the attacker<br />must find a different 24-felt entry that hashes to the same value).<br />2. `point_add` traps on entries with non-canonical X/Y limbs (`f_k1::mul` rejects<br />quotient overflow). Same second-preimage argument applies (the malformed entry<br />would have to hash to the same value as the honest one).<br />3. `adv_pipe` traps if fewer than 768 felts are queued on the advice stack.<br />A no-trap wrapper (one that returns 0 instead of trapping on bad operand-stack inputs)<br />must validate those inputs first and only call this when validation passes. See<br />`verify_prehash_native_precomp` for the canonical pattern: it multiplicatively ANDs<br />per-input validity flags and gates `exec.verify_precomp` on the combined flag.<br /><br />The scalar pointers must reference eight u32 little-endian limbs each. Non-u32 limbs trap<br />during byte peeling; the ECDSA wrapper supplies reduced scalar-field values.<br /><br />Stack transition:<br />[out_addr, u_2_addr, u_1_addr, merkle_root_addr, ...]  ->  [...]<br /><br />Local memory layout (mem[0..40]):<br />0..4   : merkle_root cache (4 felts, word-aligned)<br />4      : out_addr cache (caller-supplied accum address)<br />5      : u_1_addr cache<br />6      : u_2_addr cache<br />7      : block counter (0..32)<br />8      : u_1 byte buffer (current u32 limb, shifted right by 8 after each window)<br />9      : u_2 byte buffer<br />10     : next u32 limb index (in 0..8)<br />16..40 : entry scratch (24 felts, word-aligned at 16; first 20 are the point payload<br />consumed by point_add, last 4 are zero pad to align the hash input to 24 felts)<br /> |
