1//! Input layout definitions for ACE circuit evaluation.
2//!
3//! The layout mirrors the MASM verifier READ section: values are stored as
4//! extension-field elements in point-major order (all openings at `zeta`
5//! followed by all openings at `g * zeta`). Auxiliary trace and quotient
6//! chunk openings are provided as base-field coordinates and merged into
7//! extension elements inside the ACE circuit.
8//!
9//! All offsets and widths in this module are measured in extension-field (EF)
10//! elements. One EF element occupies two base-field elements.
11//!
12//! Besides the AIR constraints themselves, the ACE circuit computes:
13//! - Randomness expansion from `(alpha, beta)` into the full coefficient vector `[alpha, 1, beta,
14//!   beta^2, ...]`.
15//! - Auxiliary/quotient coordinate merges to recover extension-field values.
16//! - Periodic column evaluations from `z_k`.
17//! - Selector polynomial evaluations and vanishing inverses.
18//! - Lagrange-kernel weights and shifts for quotient chunk recomposition.
19//! - Constraint folding with the composition challenge and final root check.
20//!
21//! The current "stark vars" block is sufficient to derive both selector
22//! polynomials and the Lagrange-kernel weights used in quotient chunk
23//! recomposition:
24//! - Selector evaluations:
25//!   - `inv_vanishing = 1 / (z^N - 1)`
26//!   - `is_first = (z^N - 1) * inv(z - 1)`
27//!   - `is_last  = (z^N - 1) * inv(z - g^{-1})`
28//!   - `is_transition = z - g^{-1}`
29//! - Lagrange kernel inputs:
30//!   - `s0 = offset^N` and `g = subgroup_gen^N` define the shifts `s_i = s0 * g^i`.
31//!   - `weight0 = 1 / (k * s0^{k-1})` gives barycentric weights `weight_i = weight0 * g^i` for the
32//!     chunk recomposition kernel.
33//! - `z_k` is used to evaluate periodic columns inside the circuit.
34//!
35//! Layout order (both Native and Masm):
36//! 1) public_values
37//! 2) aux randomness (alpha/beta)
38//! 3) main_curr
39//! 4) aux_curr
40//! 5) quotient_curr
41//! 6) main_next
42//! 7) aux_next
43//! 8) quotient_next
44//! 9) aux_bus_boundary
45//! 10) stark_vars
46//!
47//! Notes:
48//! - `quotient_next` is included in the READ layout and is mapped via
49//!   `InputKey::QuotientChunkCoord` with `offset = 1`.
50//! - `stark_vars` reserves at least 14 EF slots for the canonical verifier inputs. Extra slots are
51//!   left available for future aux inputs.
